## 修复抽卡页显示0次的Bug（commit 3342f53）

### 现象
- 角色页显示"今日剩余8次"（通过getStudentData读原始student对象计算）
- 点进抽卡页显示0次，不让抽卡

### 根本原因
**`getDrawStatus` 缺乏跨日判断逻辑**：直接读 `remainingDraws` 字段，但：
- 当 `lastDrawDate != today` 时（今天还没任何操作），`remainingDraws` 可能是昨天抽完后的 0
- 应该在 `lastDrawDate != today` 时返回基础 3 次

**`drawGacha` 读废弃字段**：跨日重置时读 `bonusToday`（已废弃），默认为 0，导致跨日后重置为 3 而非正确值

**`gacha.js` fallback 逻辑**：`getDrawStatus` 返回 `success:false` 时不进 catch，页面次数停留在默认值；fallback 读的是废弃的 `dailyDrawLeft` 字段

### 修复方案（3个文件）
1. **`getDrawStatus`**：加 `lastDrawDate !== today` 分支，直接返回 3（今日未操作），避免读旧 remainingDraws=0
2. **`drawGacha`**：跨日重置改为 `currentRemaining = 3`（不再读废弃的 bonusToday 字段）
3. **`gacha.js`**：`getDrawStatus success:false` 时抛出异常进入 catch；fallback 优先读 `remainingDraws` 新字段；同步 globalData 时也写 `remainingDraws`

⚠️ 需重新上传云函数：getDrawStatus、drawGacha（以及之前的 useChallenge）

## 修复抽卡页一次点击抽多次+返回次数不同步（commit 059d059）

### Bug 1：点一次抽了3次（出现2个成长加速剂）
- **根因**：`onLoad` 和 `onShow` 在微信小程序进入页面时**同时触发**，导致 `loadData` 被调用两次（但这不直接导致抽多次）
- **真正根因**：`drawTap` 用 `this.data.isDrawing` 做防重复，但 `setData` 是异步的，快速点击时第一次的 `setData({ isDrawing: true })` 还没生效，第二次点击进来时 `isDrawing` 仍为 false，导致并发多次触发 `drawGacha`
- **修复**：引入同步标记 `this._isDrawing`（实例属性，立即生效），替代 `setData` 的异步 `isDrawing`
- **附带修复**：`onLoad` 初始化 `_pageLoaded = false`，`onShow` 只在 `_pageLoaded === true` 时刷新，避免进页面时重复调用

### Bug 2：返回前一页面还显示今日剩余3次（未同步）
- **根因**：`drawGacha` 成功后只更新了 globalData 的 `dailyDrawLeft`（旧字段），未更新 `remainingDraws`（新字段），而 `character.js` 读的是 `remainingDraws`
- **修复1**：`gacha.js` drawTap 成功后同时更新 globalData 的 `remainingDraws` 字段
- **修复2**：`drawGacha` 云函数返回值增加 `challengeVouchers` 和 `growthAccelerants`，确保道具数量也能同步

⚠️ 需重新上传云函数：drawGacha（必须）、useGrowthAccelerant（必须）

## drawGacha 完全重写为原子操作（commit ebe93a6）

### Bug：点一次抽卡显示"次数已用完"（并发竞态）
- **根因**：`drawGacha` 原逻辑是"先读 → 检查 → 再写"三步非原子操作，两个请求并发时都读到旧值，各自扣减，导致竞态
- **新逻辑**：
  1. 跨日重置：用普通 update 写 `remainingDraws = 3, lastDrawDate = today`
  2. **原子扣减**：`db.collection('students').where({ _id, remainingDraws: _.gt(0) }).update({ remainingDraws: _.inc(-1) })`
     - 只有 `remainingDraws > 0` 时才执行扣减
     - `updateRes.updated === 0` 表示条件不满足（次数已耗尽），直接返回失败
  3. 返回最新 `remainingDraws`
- **效果**：多个并发请求同时进来，数据库层保证只有实际剩余的次数被扣，不会重复扣减

## 彻底统一：抽卡页改用 getStudentData（commit 1c02e4f）

### 问题
角色页调 `getStudentData`（返回原始 student 对象），抽卡页调 `getDrawStatus`（云函数内部计算）。两套数据源+两套逻辑必定产生不一致。

### 修复
抽卡页 `loadData` 直接调 `getStudentData`（与角色页同源），前端用**完全相同的 JS 逻辑**计算今日剩余次数：
```js
if (lastDrawDate === today) {
  dailyLeft = student.remainingDraws
} else {
  dailyLeft = 3
}
```
两处页面现在完全一致，不再依赖任何云函数内部计算。
